3.1 注册中心概述

注册中心的主要作用

  • 动态加入
  • 动态发现
  • 动态调整
  • 统一配置

模块介绍

模块名称 模块介绍
dubbo-registry-api 包含了注册中心的所有API和抽象实现类
dubbo-registry-zookeeper 使用Zookeeper作为注册中心的实现
dubbo-registry-redis 使用redis作为注册中心的实现
dubbo-registry-default Dubbo基于内存的默认实现
dubbo-registry-multicast multicast模式的服务注册与发现

3.1.1 工作流程

注册中心的工作流程

  1. 服务提供者启动时，会向注册中心写入自己的元数据信息，同时会订阅配置元数据
  2. 消费者启动时，也会向注册中心写入自己的元数据信息，并订阅服务提供者、路由配置元数据
  3. 服务治理中心(dubbo-admin)时，会同时订阅所有消费者、服务提供者、路由配置元数据信息
  4. 当有服务提供者离开或者有新的服务提供者加入时，注册中心服务提供者目录会发生变化，变化信息会动态通知给消费者，服务治理中心
  5. 当消费方发起服务调用时，会异步将调用、统计信息等上报给监控中心(dubbo-monitor-simple)

3.1.2 数据结构

3.1.3 ZooKeeper原理概述

ZooKeeper的节点类型

  • 持久节点：服务注册后保证节点不会丢失，注册中心重启也会在
  • 持久顺序节点：在持久节点特性的基础上增加了先后顺序的能力
  • 临时节点：服务注册后连接丢失或session超时，注册节点会自动被移除
  • 临时顺序节点：在临时节点特性的基础上增加了先后顺序能力

ZooKeeper的树形结构

  1. 树的根节点是注册中心分组，下面有多个服务接口，分组值来自用户dubbo:registry中的group属性，默认/dubbo
  2. 服务接口下包含4类子目录，分别是providers,consumers,routers,configurators元数据信息
  3. 服务提供者目录（/dubbo/service/providers）下面包含的接口有多个服务者URL元数据信息
  4. 服务消费者目录（/dubbo/service/consumers）下面包含的接口有多个消费者URL元数据信息
  5. 路由配置目录（/dubbo/service/routers）下面包含多个用于消费者路由策略元数据信息
  6. 动态配置目录（/dubbo/service/configurators)下面包含多个用于服务者动态配置URL元数据信息

BeanDefinationParser的解析过程

  1. 新建SpringBean定义文件，交给Spring反射创建实例
  2. 确保没有重复定义SpringBean定义
  3. 依次尝试获取XML标签配置的name和interface 作为Bean唯一id
  4. 如果协议标签没有指定name，则默认使用dubbo
  5. 检查重复bean，如果重复则生成唯一id
  6. 每次解析向Spring注册新的BeanDefinition，后续会追加属性

    具体的标签解析

    dubbo:service

  7. 如果dubbo:service配置了class属性，那么为具体的class配置类注册Bean，并注入ref属性

attribute 提取逻辑

  1. 查找配置对象的get、set 和is前缀方法，如果标签属性和方法名称相同，则通过反射调用存储标签对应值
  2. 如果没有和get、set和is前缀方法匹配，则当作parameters参数存储，parameters是一个Map对象

注解处理逻辑

按需生成Bean

提升@Service类为Bean

ServiceAnnotationBeanPostProcessor

  1. 提取用户配置的扫描包名称
  2. 触发ServiceBean定义和注入；
  3. 指定扫描dubbo的注解@Service，不会扫描Spring的Service注解，设置过滤器
  4. 将@Service作为不同的Bean注入容器
  5. 对扫描的服务创建BeanDefinitionHolder，用户生成ServiceBean定义
  6. 注册ServiceBean定义并做数据绑定

实际使用过程中，会在@Service注解的服务中注入@Reference注解，这样就可以很方便地发起远端服务调用

为@Reference注入代理对象

5.2 服务暴露的实现原理

5.2.1 配置承载初始化

覆盖策略

  1. -D 传递给JVM参数优先级最高
  2. 代码或XML配置优先级次高
  3. 配置文件优先级最低

provider和客户端的配置

  1. 如果provider指定了配置，会透传到客户端
  2. 如果客户端也配置了相应属性，则服务端的配置会被覆盖

5.2.2 远程服务的暴露机制

服务暴露的两大部分

  1. 将持有的服务实例通过代理转换成Invoker
  2. 将Invoker通过具体的协议转换成Exporter

ref->invoker

  1. 获取配置信息到Map，用于后续构造URL
  2. 读取全局配置信息，会自动添加前缀
  3. 本地服务暴露
  4. 上报监控地址
  5. 通过动态代理转换成invoker
  6. 服务暴露后进行服务元数据注册
  7. 处理没有注册中心场景，直接暴露服务

注册中心的服务暴露 Registry#export

  1. 委托具体协议进行暴露，创建NettyServer监听端口和保存服务实例
  2. 创建注册中心对象，与注册中心创建TCP连接
  3. 注册服务元数据到注册中心
  4. 订阅configurators节点，监听服务动态属性变更事件
  5. 服务销毁收尾工作（Invoker销毁时，注销端口和map中服务实例等资源，移除已注册的元数据，去掉订阅配置监听器）

从Invoker构造拦截器链

拦截器初始化时，会自动注入ProtocolListenerWrapper和ProtocolFilterWrapper

Dubboprotocol#export

  1. 从服务分组、版本、接口和端口构造key
  2. 把exporter存储到单例DubboProtocol中
  3. 创建NettyServer，并且初始化Handler

5.2.3 本地服务的暴露机制

  1. 显式指定injvm协议进行暴露
  2. 调用injvmProtocol#export

5.3 服务消费的实现原理

4.1 扩展机制概述

4.1.1 Java Spi

实现步骤：

  1. 编写接口及实现类
  2. 在META-INF/services下创建具有接口全路径名的文件，文件内容为对应的实现类全路径名
  3. 编写调用类通过 ServiceLoader.load()获得对应的实例

4.1.2 Dubbo Spi

实现步骤

  1. 编写接口类及实现类
  2. 在META-INF/services、META-INF/dubbo或者META/dubbo/internal下创建具有接口全路径名的文件，文件内容为一个键值对（对应实现实例名=对应的实现类全路径名）
  3. 编写调用类 通过ExtensionLoader.getExtensionLoader(接口名).getDefaultExteion()获得对应实例

Java SPI 和 Dubbo SPI的比较

  1. 实例化，JavaSPI会一次性实例化所有实现，但是DubboSPI则会加载类但是会按需实例化实现
  2. 扩展失败，JavaSPI如果扩展失败，则无法获得扩展名称
  3. DubboSPI增加了对于IoC和AOP的支持，一个扩展可以直接setter注入其他扩展

4.1.3 Dubbo 的配置规范

规范名 规范说明
SPI配置文件路径 META-INF/services/,META-INF/dubbo/,META-INF/dubbo/internal/
SPI配置文件名称 全路径类名
文件内容格式 key=value，换行符分隔

4.1.4 扩展点的分类与缓存

扩展类的分类

按照内容分类

  1. Class缓存
  2. 实例缓存

按照扩展类种类分类

  1. 普通扩展类
  2. 包装扩展类：通用抽象，构造方法中会传入一个具体的实现
  3. 自适应扩展类：抽象扩展接口，具体实现在运行时由传入参数动态确定
  4. 其他缓存：如扩展类加载器缓存，扩展名缓存

扩展类缓存

集合名 缓存类型
Holder<Map<String,Class<?>>> cachedClasses 普通扩展类缓存，不包括自适应扩展类和Wrapper类
Set<Class<?>> cachedWrapperClasses Wrapper类缓存
Class<?> cachedAdaptiveClass 自适应扩展类缓存
CocurrentMap<Class<?>,String> cachedNames 扩展类与扩展名缓存
CocurrentMap<Class,ExtentionLoader> EXTENSION_LOADERS 扩展类与对应的扩展类加载器缓存
CocurrentMap<Class<?>,Object> EXTENSION_INSTANCES 扩展类与类初始化后的实例
Map<String ,Activate> cachedActivates 扩展名与@Activate的缓存

  2. 自动加载：setter注入
  3. 自适应：@Adaptive参数匹配
  4. 自动激活：

4.2 扩展点注解

4.2.1 扩展点注解 @SPI

  1. 使用范围： 可以用在类，接口，枚举类上
  2. 作用：标记SPI接口，运行时通过配置找到对应的实现类

4.2.2 扩展点自适应注解 @Adaptive

  1. 使用范围：可以使用类，接口，枚举类和方法上
  2. 作用：
    1. 标记在类级别上：直接将整个类作为默认实现，不再自动生成匹配代码
    2. 标记在方法上：（会生成匹配代码，在匹配代码中）根据传入的指定Contant.key从url中提取参数进行匹配，最终生成指定的实现类

4.2.3扩展点自动激活注解 @Activate

Active参数

参数名 效果
String[] group() URL中的分组如果匹配则激活，则可以设置多个
String[] value() 查找URL中如果含有该key值，则激活
String[] before() 填写扩展点列表，表示哪些扩展点在本扩展点之前
String[] after() 表示哪些扩展点在本扩展点之后
int order() 整型，直接的排序信息

4.3 ExtensionLoader的工作原理

4.3.1 工作流程

getExtension(String name)的工作流程

加载中的每一步都会先检查缓存中是否已经存在所需的数据

  1. 框架读取SPI对应路径下的配置文件，根据配置加载所有扩展类并缓存（不初始化）
  2. 根据传入的名称初始化对应的类
  3. 尝试查找符合条件的包装类
  4. 返回对应的扩展类实例

getAdaptiveExtension()的工作流程

  1. 加载配置文件
  2. 生成自适应的类的代码字符串
  3. 获取类加载器和编译器，并用编译器编译刚才生成的代码字符串
  4. 返回对应的自适应实例

4.3.2 getExtension的工作原理

injectExtension()利用setter实现了一个简单的IoC，完成依赖注入

4.3.3 getAdaptiveExtension的实现原理

getAdaptiveExtension的代码生成步骤

  1. 生成package、import 、类名称等头部信息
  2. 遍历接口所有方法，获取方法的返回类型、参数类型、异常类型等
  3. 生产参数为空的空校验代码
  4. 生成默认实现类的名称
  5. 生成获取扩展点名称的代码
  6. 生成获取具体扩展实现类的代码
  7. 生成调用结果代码

由以上自动代码产生的效果：
如果 SPI上备注了名称，Adaptive上也备注了名称，则会将SPI的名称作为默认名称，优先以Adaptive名称查找实现类，失败则使用SPI名称

4.3.4 getActiveExtension的实现原理

  1. 检查缓存，如果缓存中没有，则初始化所有扩展类实现的集合
  2. 遍历@Activate集合，然后根据URL传入的匹配条件（group，name）等，得到所有符合激活条件的扩展类实现
  3. 遍历所有用户自定义扩展的名称，根据用户URL配置的顺序，调整扩展点激活顺序
  4. 返回所有自动激活类姬和

如果URL参数中传入了-default，则所有默认的@Activate都不会被激活，只有URL参数中指定的扩展点会被激活

4.3.5 ExtensionFactory的实现原理

  1. Dubbo和Spring如何被打通？
    通过Spring的上下文，在SpringExtensionFatory中提供了保存Spring的上下文的静态方法
  2. AdaptiveExtensionFactory 在构造方法中缓存了所有的扩展类工程，并通过TreeSet排序，SPI在前，Spring在后

4.4 扩展点动态编译的实现

4.4.1总体结构

Dubbo中的三种编译器

  1. JDK编译器
  2. Javassist编译器
  3. AdaptiveCompiler编译器

配置

  1. @SPI注解的默认的编译器是javassist；想改变默认编译器，可以通过<dubbo:application compiler=”jdk”>进行配置
  2. @Adaptive注解的默认编译器是AdaptiveCompiler

AdaptiveCompiler的主要抽象逻辑

  1. 通过正则匹配出包路径、类名，再根据包路径、类名拼接处全路径类名
  2. 尝试通过Class.forName加载该类并返回，防止重复编译
  3. 调用doCompile进行编译

4.4.2 Javassist动态代码编译

Javassist动态编译步骤

  1. 初始化Javassist，设置默认参数
  2. 通过正则匹配出所有import的包，并使用Javassist添加import
  3. 通过正则匹配出所有extends的包，创建Class对象，并使用Javassist添加extends
  4. 通过正则匹配出所有implements的包，并使用Javassist添加implements
  5. 通过正则匹配出类面的所有内容，即得到{}中的内容，再通过正则匹配出所有方法，并使用Javassist添加类内容
  6. 生成Class对象

4.4.3 JDK动态代码编译

主要流程：

  1. 初始化JavaFileObject对象，传入代码字符串，
  2. 调用JavaCompiler.CompilationTask编译出具体的类
  3. JavaFileMananer负责管理类文件的输入和输出位置

接口与类介绍

  1. JavaFileObject接口。字符串代码被包装成一个文件对象，并提供二进制流的接口
  2. JavaFileManager接口。主要管理文件的输入输出位置。
  3. JavaCompiler.CompilationTask把JavaFileObject编译成具体的类
